# LucaWriter 知识库引擎改造方案

> 目标读者：负责落地的工程 AI（DeepSeek）。本文档自包含，不依赖任何对话上下文。
> 落地原则：**全程向后兼容、增量迁移、不重通读、不动 COO 对外契约**（见 §1）。
> 一句话定位：现在的 schema 很扎实，瓶颈在「实体归并」和「喂给 AI 的记忆」两层。**这是升级，不是重写。**

---

## 0. 现状审计（先看清楚现在有什么）

### 0.1 存储结构

- 每本书一个 SQLite（`BOOKS_DIR/<book_id>/kb.db`）；COO v2 作品级共享库 = 一个 work 级 SQLite（`WORKS_DIR/<work_id>/kb.db`），章节用复合 ID `"{book_id}::{chapter_id}"` 落到同一个库。下文统一把 `book_id`/`work_id` 叫 **库 ID**。
- 表定义在 [`backend/kb_storage.py:61-270`](backend/kb_storage.py)：
  `chapters / entities / mentions / events / foreshadowing / rules / rule_mentions / timeline_event_meta / timeline_relations / consistency_alerts / rt_state / rt_logs / embedding_chunks / vector_entries / kb_proposals / kb_edit_log`。
- 向量：`vector_entries`（`embedding BLOB + dimensions + document + metadata_json`），检索靠 [`embed_query`](backend/kb_storage.py:978) 在 Python 里 **numpy 暴力余弦**（O(N) 全表扫描）。嵌入后端 [`get_embedding_backend`](backend/embeddings.py:184)：本地 `BAAI/bge-small-zh-v1.5` 或 API。

### 0.2 关键管线（带行号，落地时核对）

| 能力 | 入口 | 现状 |
|---|---|---|
| 单本通读 | [`do_readthrough`](backend/kb_pipeline.py:705) | 逐章/批量读，写库 |
| 作品级通读 | [`do_readthrough_work`](backend/kb_pipeline.py:1234) | 按阅读线遍历全部子书章节+设定+卷边界，写 work 库 |
| 章节结构化抽取 | [`ai_read_chapter_structured`](backend/kb_pipeline.py:157) | 出 entities/events/foreshadowing/rules JSON；**已支持二周目纠错**（`prior_records`，prompt 见 :175-182） |
| 落库 | [`apply_structured_result`](backend/kb_pipeline.py:418) | **先 `delete_chapter_artifacts` 整章清空再重插** |
| 实体去重 | [`upsert_entity`](backend/kb_storage.py:351) | **仅按 `canonical_name` 精确匹配**；别名存了但插入路径不参与去重 |
| 喂给 AI 的前情 | [`_build_prev_context`](backend/kb_pipeline.py:679) | **固定 6000 字**：最近 8 章摘要 + 40 个实体（**按拼音字母序**取前 40） |
| 建向量 | [`incremental_embed`](backend/kb_pipeline.py:573) | 在通读循环**全部结束后**才跑（work 管线 :1443）；读的过程中无语义检索 |
| 增量同步 | [`sync_work_chapters`](backend/kb_pipeline.py:1112) + [`_do_work_sync_kb`](backend/main.py:11042) | 只重读 `content_hash` 变化的章，带旧记录纠错。**但靠用户手动点「同步」** |
| 吃书雷达 | [`detect_consistency`](backend/kb_pipeline.py:2034) | 写作时实时检查冲突 → `consistency_alerts`；**只提醒不改库**，且不在通读期跑 |
| 写后调度 | [`_schedule_kb_after_write_jobs`](backend/main.py:11252) | 章节写完后调度摘要/时间线/预言 |

### 0.3 五个真实缺口（本方案要补的）

1. **模糊匹配缺失** → 同一角色随 AI 命名漂移裂成多个实体（`upsert_entity` 只认精确串；`UNIQUE(book_id,canonical_name)` 还会把两个真同名实体错并）。
2. **AI 记忆是死窗口** → 读到第 1000 章时对前 990 章的认知 ≈ 最近 8 章摘要 + 字母序 40 实体。长程理解断裂，跨百章伏笔连不上。
3. **纠错无版本、无跨章矛盾消解** → reread 是「整章删了重插」，没有「v1 说 X、重读改成 Y」的历史，也不在通读期主动发现并修正矛盾。
4. **增量更新靠手点** → 没有「保存即后台更新」。
5. **检索是暴力扫描** → 单查询 O(N)，大长篇下若每章都检索会变慢（需 ANN 兜底）。

## 1. 落地总原则（每个 PR 都必须守）

1. **不重通读、不清旧库**：用户的 `kb.db` 是花大量时间通读出来的核心资产。所有新表用 `CREATE TABLE IF NOT EXISTS`，所有新列用「检测后 `ALTER`」，迁移幂等。**严禁** `DROP` / 清空已有 `entities/mentions/events/...`。只有用户明确点「重通读/清空」才允许重建。
2. **Schema 版本用 `PRAGMA user_version`**：在 [`init_db`](backend/kb_storage.py) 里读 `user_version`，按版本号跑增量迁移函数，跑完写新版本号。不要用「表是否存在」做隐式判断之外的脏检测。
3. **COO 对外契约尽量不动**：新表都落在 `ai/kb.db` 内部，对 Coobox 是不透明 blob。**只新增表/列，绝不重命名或删除已有表/列** → Coobox 端无需改。若确实改了 `.coo` 的目录结构或 `manifest`，才需按 CLAUDE.md「三处同步」改 [`backend/main.py`](backend/main.py) 的 `_build_coo_zip`/`_import_coo_zip`、`cooverter.py`、Coobox 解析验签。**本方案默认不触发三处同步**（落地时确认 Coobox 不对 kb.db 做严格 schema 校验，只读它需要的表）。
4. **特性开关**：每个模块都挂 settings 开关（默认值见各模块），方便灰度与回退。
5. **嵌入后端可缺省降级**：本地模型/向量库不可用时，所有新逻辑必须回落到现有行为，不能让通读崩溃。

## 2. 模块 A：实体归并 / 模糊匹配（最高杠杆，先做）

**问题**：[`upsert_entity`](backend/kb_storage.py:351) 只做 `canonical_name` 精确匹配。

**目标**：新实体进来时，能识别出它其实是已有实体（哪怕换了称呼），自动归并；同时避免把两个真不同的同名实体并错。

### 2.1 新增数据结构

```sql
-- 别名反查表：alias_norm → entity_id（归一化后的别名/称呼）
CREATE TABLE IF NOT EXISTS entity_aliases (
  id          TEXT PRIMARY KEY,
  book_id     TEXT NOT NULL,          -- 库 ID（work 或 book）
  entity_id   TEXT NOT NULL,
  alias_norm  TEXT NOT NULL,          -- 归一化后的别名
  alias_raw   TEXT NOT NULL,          -- 原始别名
  source      TEXT NOT NULL DEFAULT 'ai',  -- ai|user|merge
  created_at  INTEGER NOT NULL,
  FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_entity_aliases_lookup ON entity_aliases(book_id, alias_norm);

-- 归并审计（可逆）：记录每次软合并，便于回退与编辑历史
CREATE TABLE IF NOT EXISTS entity_merges (
  id          TEXT PRIMARY KEY,
  book_id     TEXT NOT NULL,
  from_name   TEXT NOT NULL,          -- 被并入的原 canonical
  into_id     TEXT NOT NULL,          -- 合并目标 entity_id
  reason      TEXT,                   -- alias|embedding|llm|user
  similarity  REAL,
  created_at  INTEGER NOT NULL
);
```

并给 `entities` 增列（`ALTER` 幂等）：

```sql
ALTER TABLE entities ADD COLUMN name_norm   TEXT DEFAULT '';   -- canonical 的归一化键
ALTER TABLE entities ADD COLUMN mention_cnt INTEGER DEFAULT 0; -- 提及计数，供重要性排序（§3）
```

### 2.2 归一化函数（新增 `kb_storage.normalize_name`）

```python
def normalize_name(s: str) -> str:
    # 1) NFKC（全角→半角、兼容字符规整）  2) strip  3) 去成对/装饰标点与空白
    # 4) 拉丁字母 lower  5) 去常见敬称尾缀（"老/小/阿"前缀与"君/氏/大人/先生"后缀按需，谨慎）
    # 中文不做分词，仅做字符级归一，避免误并。
```

> 注意：归一化要**保守**。宁可少并（留给嵌入/LLM 兜底）也不要把「赵云」和「赵子龙」用规则强并——那交给别名表与嵌入。

### 2.3 解析主流程（新增 `resolve_entity`，替换 `upsert_entity` 的直接调用）

调用点：[`apply_structured_result`](backend/kb_pipeline.py:418) 与 [`apply_partial_structured_result`](backend/kb_pipeline.py:2244) 里的 `upsert_entity(...)` 改为 `resolve_entity(...)`。

解析顺序（命中即停）：

1. **精确**：`canonical_name` 命中现有 `entities.canonical_name` → 该实体。
2. **别名反查**：`normalize_name(incoming_canonical)` 或任一 `incoming_alias` 的归一化 命中 `entity_aliases.alias_norm` → 该实体。
3. **嵌入近邻**：对 `name + type + 首条 fact` 取嵌入，查 `vector_entries`（`where={'source_type':'entity'}`）top-3；若 `similarity ≥ τ_high(默认 0.86)` **且 type 相容**（见 2.4）→ 合并候选。
4. **边界态 LLM 兜底**：`τ_low(0.78) ≤ sim < τ_high` 进入批量消歧队列（见 2.5），先按「新建」落库，待批量裁决后再决定并/留。
5. 否则 **新建实体**，写 `name_norm`，把 canonical 与所有别名写入 `entity_aliases`。

合并动作 `merge_entities(into_id, from_id|from_payload)`：
- mentions 全部 repoint 到 `into_id`；aliases 取并集；`first_chapter_id` 取更早；`mention_cnt` 累加；
- 在 `entity_merges` 记一条（reason/similarity）；
- **不硬删历史审计**：被并实体行可删，但 merge 记录与 alias 必须保留，保证可回溯/可在编辑历史里展示。

### 2.4 type 相容规则

- 完全相同 type → 相容。
- 一方为 `未分类/未知` → 相容（吸收到更具体的一方）。
- 维护一张小的同义类映射（如 `势力≈组织≈阵营`、`地点≈场所`），落在常量里，可被 settings 覆盖。
- 类明显冲突（`人物` vs `地点`）→ 即使名字相同也**不并**，反而提示可能同名异物。

### 2.5 LLM 批量消歧（避免每个实体一次 AI 调用）

- 把边界态候选攒成一批（如每 N=12 对，或每章末），一次 prompt 让模型判断「下列名字对是否指同一实体」，返回 `[{a,b,same:bool,confidence}]`。
- 仅在 `confidence ≥ 0.7` 时执行合并；写 `entity_merges(reason='llm')`。
- 该步骤可被 settings `entity_resolution_llm=false` 关掉，仅留规则+嵌入。

### 2.6 迁移与回填

- 迁移脚本：遍历现有 `entities`，回填 `name_norm`；把每个实体的 `canonical_name + aliases(JSON)` 拆进 `entity_aliases`；`mention_cnt` 用 `COUNT(mentions)` 回填。
- **不**自动合并历史已裂实体（避免一次性误并）。提供一个**手动/可选**的「一键归并体检」后台任务：扫描跑 2.3 的 2/3/4 步，产出**合并建议清单**（写进 `kb_proposals`，复用现有提案/编辑日志 UI），由用户确认后再合并。这条线尊重「旧库是资产」。

### 2.7 验证

- 构造测试：实体 A 先以「李云」入库，再以「李公子」（type=人物，含 fact 重叠）入库 → 应归并为 1 个实体、2 条别名、mentions 合并。
- 反例：两个「老王」（type=人物，facts 明显不同、嵌入相似度低）→ 保持 2 个实体，不误并。
- 回归：现有库迁移后 `entities` 行数不变、`entity_aliases` 有回填、通读不报错。

## 3. 模块 B：检索式阅读上下文（解锁大长篇）

**问题**：[`_build_prev_context`](backend/kb_pipeline.py:679) 固定 6000 字 / 最近 8 摘要 / 字母序 40 实体。

**目标**：读某一章时，喂给 AI 的是**与当前章真正相关**的历史知识，而不是一个固定小抄。

### 3.1 边读边嵌入

- 把 [`incremental_embed`](backend/kb_pipeline.py:573) 的「实体/事件/伏笔/摘要 → chunk」逻辑提取成可对**单章增量**调用的 `embed_chunks_for_chapter(book_id, chapter_id, settings)`。
- 在通读循环里，每章 `apply_structured_result` 之后立即增量嵌入该章产物（而非等全书读完）。`queue_chunk` 已是「按 content_hash 去重、只嵌变化」的设计，天然增量，直接复用。
- 全书末尾仍跑一次 `incremental_embed` 收尾（补漏 + `prune_vector_entries`）。

### 3.2 新的上下文装配 `build_reading_context`

替换通读三处对 `_build_prev_context` 的调用（[:853](backend/kb_pipeline.py:853)、[:1181](backend/kb_pipeline.py:1181)、[:1405](backend/kb_pipeline.py:1405)）。

```python
def build_reading_context(book_id, chapter_text, settings, token_budget):
    # 预算 = 有效上下文长 * reading_ctx_fraction(默认 0.35)
    # ---- 固定锚（pinned，必进）----
    #   - 最近 3 章摘要
    #   - 所有 status='open' 的伏笔（按 hint_chapter 倒序，封顶 ~15 条）
    #   - 重要实体 top-N（按 mention_cnt desc, updated_at desc，封顶 ~15）
    # ---- 动态检索（dynamic，填满剩余预算）----
    #   - 用 chapter_text 前 ~1500 字做 query，embed_query top_k≈24
    #   - 跨 entity/event/foreshadowing/chapter_summary，去重掉已在 pinned 的
    #   - 命中实体时，渲染其 render_entity_block（已有，:1562）截断版
    # 组装、按预算截断、回落：嵌入不可用 → 调旧 _build_prev_context
```

- **重要性排序**：实体一律按 `mention_cnt`（§2.6 已回填、§2.3 合并时累加）排序，杜绝字母序。
- `embed_query` 已支持 `where` 元数据过滤（[:978](backend/kb_storage.py:978)），可分桶取（如保证至少 N 条 entity、N 条 event）。

### 3.3 性能：暴力扫描的兜底

- `embed_query` 是 O(N) 全表 numpy。每章一次 query，1700 章 × 数万向量在后台尚可接受，但要：
  - 给 query 向量与 `vector_entries` 的矩阵化点积（一次 `np.vstack` 批量算，不要逐行 Python 循环）——**这是必须做的小优化**。
  - 进程内缓存「本次通读已加载的向量矩阵」，章节间复用，仅在新增 chunk 时增量拼接。
- **可选 Phase（不阻塞主线）**：引入 ANN（`hnswlib` 或 `sqlite-vss`）做近邻；接口保持 `embed_query` 签名，内部可切换，缺库回落暴力。settings `vector_index=brute|hnsw`。

### 3.4 验证

- 在一部 ≥200 章的测试作品上：读第 N 章时打印装配出的 context，应包含与该章人物/地点相关的历史事实，而非字母序前 40。
- 计时：单章 context 装配（含 1 次 embed + 1 次 query）在本地后端应 < 1.5s（矩阵化后）。
- 回落：关掉嵌入后端，通读仍跑通（走旧窗口）。

## 4. 模块 C：自动纠错 + 事实版本化

**问题**：reread 是 `delete_chapter_artifacts` 整章删重插（[:419](backend/kb_pipeline.py:419)），没有历史、没有跨章矛盾消解。

**目标**：多次重读能**纠正**而非**覆盖**，保留「这个事实改过几次、为什么改」，并在通读期主动发现矛盾。

### 4.1 事实版本表

```sql
CREATE TABLE IF NOT EXISTS fact_revisions (
  id           TEXT PRIMARY KEY,
  book_id      TEXT NOT NULL,
  entity_id    TEXT,
  chapter_id   TEXT NOT NULL,
  fact         TEXT NOT NULL,
  snippet      TEXT,
  confidence   REAL DEFAULT 0.6,
  pass_no      INTEGER DEFAULT 1,         -- 第几遍读得到的
  status       TEXT NOT NULL DEFAULT 'active', -- active|superseded|rejected
  supersedes   TEXT,                      -- 被本条取代的旧 revision id
  reason       TEXT,                      -- 纠错说明
  created_at   INTEGER NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_fact_rev_entity ON fact_revisions(book_id, entity_id, status);
```

给 `mentions` 增列：`confidence REAL DEFAULT 0.6`、`pass_no INTEGER DEFAULT 1`。`mentions` 保存「当前最优」，`fact_revisions` 保存全历史。

### 4.2 reread 由「删重插」改为「diff + 版本化」

改 [`apply_structured_result`](backend/kb_pipeline.py:418)：当本章 `existing.status=='done'`（即重读）时，**不**整章 `delete_chapter_artifacts`，改为：

1. 取本章旧 mentions/events/...；
2. 对模型新产物做匹配（fact 文本归一 + 嵌入相似）：
   - 旧有新也有（基本一致）→ 保留，`pass_no+1`，`confidence` 上调；
   - 旧有新无 → 标 `status='superseded'`（不硬删，进 `fact_revisions`）；
   - 旧无新有 → 新增 + 写 revision；
   - 冲突（同实体同属性，值不同）→ 旧条 `superseded`、新条 `active(supersedes=旧id)`、`reason` 记模型给的纠错说明，并发一条 `consistency_alerts(kind=continuity)`。
3. 首次读（无旧记录）走原路径（可继续删重插或直接插）。

> 模型侧已具备纠错能力：[`ai_read_chapter_structured`](backend/kb_pipeline.py:157) 在 `prior_records` 非空时走「二周目纠错」prompt（:175-182）。**扩展该 prompt 的输出 schema**，让它显式返回 `corrections:[{old_fact_hint, new_fact, reason, confidence}]`，供 4.2 精确定位要 supersede 的旧条目（而不是靠后端猜 diff）。

### 4.3 通读期矛盾发现（把吃书雷达接进来）

- 现在 [`detect_consistency`](backend/kb_pipeline.py:2034) 只在写作时跑。新增：通读每章落库后，对**高置信旧事实**与**本章新事实**做一次轻量冲突检查（可纯结构化比对 + 必要时调一次 AI），命中写 `consistency_alerts`。
- 不自动改库（尊重作者裁判），但把冲突章**入队**给模块 D 的「纠错 pass」复读。

### 4.4 多遍精读（pass 调度）

- 新增后台任务 `kb-refine-pass`：挑选 `confidence < 阈值` 的实体/事实、或有 open 矛盾的章，带**完整检索上下文（§3）+ prior_records** 重读，产出新 revision、抬升置信度。
- 受 settings `auto_refine_passes`（默认 1）控制最多自动复读几遍；每遍只动「可疑」章，不全量重跑。

### 4.5 验证

- 改一章正文里某人物的设定（如「左撇子」→「右撇子」）→ 触发重读 → `mentions` 当前值更新为右、`fact_revisions` 存有 superseded 的「左」、且生成一条 continuity alert。
- 旧库迁移后 `fact_revisions` 为空、`mentions.confidence` 默认 0.6，通读/同步不报错。

## 5. 模块 D：后台自动更新（增量调度）

**问题**：增量同步 [`sync_work_chapters`](backend/kb_pipeline.py:1112) 已能「只重读变更章+纠错」，但靠用户手点「同步」。

**目标**：保存/导入/合并 → **自动**在后台增量更新知识库，去抖、可断点续、不打扰、可关。

### 5.1 脏队列

```sql
CREATE TABLE IF NOT EXISTS kb_dirty_queue (
  id          TEXT PRIMARY KEY,
  work_id     TEXT NOT NULL,
  book_id     TEXT NOT NULL,
  chapter_id  TEXT NOT NULL,
  reason      TEXT,                  -- save|import|merge|lore|refine
  enqueued_at INTEGER NOT NULL,
  status      TEXT NOT NULL DEFAULT 'pending', -- pending|processing|done|error
  attempts    INTEGER DEFAULT 0,
  UNIQUE(work_id, book_id, chapter_id)   -- 同章去重，重复保存只留一条
);
CREATE INDEX IF NOT EXISTS idx_dirty_pending ON kb_dirty_queue(work_id, status, enqueued_at);
```

> 队列落库（不是内存），**进程重启后能续**。这是「后台自动」的可靠性关键。

### 5.2 入队点

- 章节保存接口（写 chapter 文件处）：若 `content_hash` 相对 kb 里记录有变 → `enqueue(work_id, book_id, cid, 'save')`。复用现有变更检测逻辑（[`main.py:5366-5382`](backend/main.py) 的 hash 比对）。
- 导入新卷 / merge-coo / lore 编辑 → 批量入队。
- 入队同时更新该 work 的 `last_edit_at`（settings 表或单独小表）。

### 5.3 调度器（一个全局守护线程）

```
loop（每 ~15s 醒一次）:
  for work in works_with_pending:
    if 该 work 正在跑用户发起的通读(rt_state.status=='running')  -> skip
    if now - last_edit_at(work) < idle_delay(默认 45s)            -> skip  # 去抖
    取该 work 的 pending 章（批量，封顶 N）-> 标 processing
    复用 sync_work_chapters 跑增量重读纠错（带 §3 检索上下文、§4 版本化）
    成功 -> done；失败 -> attempts+1，超 3 次标 error 并告警
    跑完触发 §3.1 增量嵌入 + render_markdown_views
```

- **去抖**：连续打字不会每次都跑；停手 45s 后才动。
- **互斥**：用户手动通读优先，自动任务让路；用 `rt_state` + 线程名（`kb_auto_<work_id>`）防重入。
- **节流**：自动 pass 用较小并发/批量，避免抢占写作时的 AI 配额。
- **开关**：settings `auto_kb_update`（默认 on）、`auto_kb_idle_delay`、`auto_kb_max_passes`。关掉后退回纯手动「同步」。
- **启动恢复**：进程启动时扫 `kb_dirty_queue` 把 `processing` 复位为 `pending`，继续消费。

### 5.4 与现有「写后调度」的关系

- [`_schedule_kb_after_write_jobs`](backend/main.py:11252) 负责摘要/时间线/预言，保持不变；模块 D 是**知识库正文层**的自动增量，二者并行不冲突（注意别对同一章同时跑两个会写 mentions 的任务——用 `kb_auto_<work_id>` 串行化知识库写入）。

### 5.5 验证

- 编辑一章 → 45s 内不动 → 之后后台自动重读该章、`kb_dirty_queue` 该行变 done、`mentions` 更新、UI 进度/日志有体现（§6）。
- 连续编辑 5 章 → 合并为一批、不抖动、不重复跑。
- 杀进程再起 → 未完成的 pending 继续被消费。
- `auto_kb_update=false` → 不自动跑，手动「同步」仍工作。

## 6. 模块 E：通读面板可观测性修复（原始 bug，体量小但要做）

> 这部分是用户最初反馈的具体 bug，独立于上面引擎升级，**可优先快速落地**。位置：元数据 Modal 右侧「章节通读进度 / 通读日志」面板。

### 6.1 前端轮询回调丢失（致命）

- `api(method,path,data,cb)` 签名见 [`index.html:2129`](frontend/index.html)。
- [`_startWorkRTPoll`](frontend/index.html:3610) 写成 `api('GET', url, function(d,s){...})` —— 回调被当成第 3 个参数 `data`，`cb` 为 `undefined`，**整个轮询回调从不执行** → 日志不更新、笔记不加载、进度不刷新、done 不收尾。
- 修复：改成 4 参 `api('GET', url, undefined, function(d,s){...})`。同样的 3 参误用还在 [:3454](frontend/index.html:3454) 和 [:3635](frontend/index.html:3635)，一并修。

### 6.2 work 级状态缺 `kb_overview`

- work 通读状态接口 [`main.py:3967-4027`](backend/main.py) 没有返回 `kb_overview`（book 级在 [:4507](backend/main.py) 有）。
- 修复：work 管线把一切写进 work 库（复合 ID），所以 `kb_storage.get_kb_overview(wid, ...)` 可直接用。在 work 状态响应里补 `resp['kb_overview'] = kb_storage.get_kb_overview(wid, int(resp.get('current_idx') or -1))`。
- 另：work 级日志/笔记本就该来自 work 库（管线 `rt_log(work_id,...)` / 写 work 库），现状逻辑正确，**真正卡点是 6.1 的回调 bug**。

### 6.3 空章节跳过不计入进度

- [`do_readthrough_work`](backend/kb_pipeline.py:1384-1386)：空章节 `rt_log(...'跳过空章节')` 后直接 `continue`，**不写状态**，于是 [`_work_detail`](backend/main.py:2261) 算 `reread_status` 时该章永远 `unread`，进度条永不满。
- 修复：跳过空章时 `upsert_chapter(work_id, storage_id, idx=ch_idx, title=ch_title, status='skipped', content_hash=hash_content(content))` 并 `chapter_count += 1`。
- 配套：
  - [`_work_detail`](backend/main.py:2287)：把 `w_status in ('done','skipped')` 都视为已处理（`reread_status` 给 `'skipped'` 或复用 `'unchanged'`）。
  - [`get_done_chapter_count`](backend/kb_storage.py:1040)：计数 `status IN ('done','skipped')`，让「已读 N/M」能到达 M。
  - 前端进度网格（[:3534/:3541/:3672](frontend/index.html)）：非 `unread` 即算已处理，可给 `.skipped` 段加一个更淡的样式以区分（仅改颜色/opacity，遵守 CLAUDE.md：禁 transform/blur/box-shadow）。

### 6.4 日志区重写为「真日志」

- 现状「通读日志」其实是一个伪人格聊天气泡（`rt-dialogue` + 头像 L + `rtStream`「我正在读《…》」），真正的 `rtLogs` 列表在下面却（因 6.1）从不填充。
- 重写（[`index.html:1712-1729`](frontend/index.html)）：
  - 删掉 `rt-dialogue / rt-avatar / rt-message / rtStream / rtSystemLine / rtThinking` 这套人格气泡。
  - 顶部留一行**紧凑状态条**：`阶段 · 已读 N/M · running|paused|done`（数据来自状态接口 `phase/done_count/total/status`）。
  - 主体是真实滚动日志 `rtLogs`：时间戳 + 后端 `recent_logs`（`rt_log` 写的「完成 (3/100) 《书名》章名」「跳过空章节」「重读纠错」「统一建立检索索引」等，本就是真事件）。新行追加、自动滚到底。

### 6.5 底部「全书笔记」→ 知识库规模可视化云图

- 现状 `#rtSource` textarea（placeholder「通读完成后此处显示全书笔记…」）对 work 级基本是空的且无信息量。**替换**为非交互的「知识库索引云图」，让人**看得见后台在建索引**。

## 7. 统一数据库迁移清单

在 [`init_db`](backend/kb_storage.py) 末尾按 `PRAGMA user_version` 跑：

```
v(current) -> v+1: 模块 A —— entity_aliases / entity_merges 建表；entities 加 name_norm/mention_cnt；回填 name_norm + 拆别名 + mention_cnt
v+1 -> v+2: 模块 C —— fact_revisions 建表；mentions 加 confidence/pass_no
v+2 -> v+3: 模块 D —— kb_dirty_queue 建表
```

- 全部 `CREATE TABLE IF NOT EXISTS` / 列存在性检测后 `ALTER`，幂等可重入。
- 迁移**只加不减**，旧数据原样保留。
- 迁移失败要**不致命**：包 try/except + 日志，失败回落到「无新表则走旧逻辑」。

## 8. 分阶段路线（每阶段独立可上线、可回退）

| 阶段 | 模块 | 改动面 | 验证关口 | 风险 |
|---|---|---|---|---|
| **P0** | §6 可观测性 | 前端 + 2 个后端小改 | 6.6 | 极低，纯展示/计数 |

| **P2** | §3 检索上下文 | kb_pipeline 装配 + 边读边嵌 | 3.4 | 中（性能）→ 矩阵化 + 缓存 + ANN 兜底 |
| **P3** | §4 版本化纠错 | apply_* 改 diff + 迁移 v+2 + prompt 扩展 | 4.5 | 中（diff 逻辑）→ 首读仍走旧路径 |

## 9. 风险与回退

- **误并实体**：阈值保守（τ_high=0.86 起，按真实数据调）、type 必须相容、`entity_merges` 可逆、历史库走「提案体检」由人确认。任何自动合并都能在编辑日志看到并撤销。
- **检索性能**：暴力余弦必须矩阵化 + 进程内缓存；大长篇上 ANN 兜底。任何嵌入失败回落旧窗口，绝不让通读崩。
- **自动更新抢占**：用户手动通读永远优先；自动任务串行化、去抖、可一键关。
- **COO 兼容**：只加表/列，不改对外结构 → Coobox 不需动。若某天确需改 `.coo` 结构，再按 CLAUDE.md 三处同步（`main.py` 的 `_build_coo_zip`/`_import_coo_zip`、`cooverter.py`、Coobox 验签）。
- **旧库资产**：全程不清不重通读；所有重活儿都是「变更章增量 + 版本追加」。

## 10. 能撑多大？（落地后的预期）

- **现状天花板**：约「三体」三部级（人物可控、跨度有限）。
- **做完 P1+P2**：实体不再裂、AI 读每章看到的是相关历史而非死窗口 → 可稳到「全职高手」(~1700 章) / 「银河帝国」式多卷长跨度。
- **做完 P3+P4**：多遍自动纠错 + 保存即后台增量 → 作品越写越准，且作者无感。
- 存储从来不是瓶颈（SQLite 千万行无压力）；瓶颈一直是「实体归并 + 喂给 AI 的记忆」这两层——本方案正是冲它们去的。
